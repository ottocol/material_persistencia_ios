Backend As A Service

Aunque hay algunas aplicaciones que pueden funcionar por sí solas sin necesidad de un servidor, muchas aplicaciones actuales necesitan de un backend para poder almacenar datos “en la nube”. Así el usuario pueda acceder a su información independientemente del dispositivo que esté usando. Otras muchas aplicaciones lo usan para que se pueda compartir información con otros usuarios.

El problema es que desarrollar un backend para una aplicación móvil no es una tarea trivial, incluso si se tienen los conocimientos necesarios. Hay que escribir la aplicación en el lado del servidor, desplegarla, alojarla en algún hosting, hacerla escalable, …. Y además muchos desarrolladores de aplicaciones móviles no tienen las habilidades necesarias, ya que en el backend se usan lenguajes, tecnologías y herramientas totalmente diferentes de los usados para el desarrollo del cliente.

Esto ha hecho que surja un mercado de lo que se conoce como “Backend as a Service” (o BaaS), es decir, ofrecer como servicio “listo para usar” las funcionalidades más típicas de los backends para aplicaciones móviles: autentificación de usuarios, persistencia de datos “en la nube”, analíticas de tráfico, notificaciones push etc. Por supuesto las propias plataformas de desarrollo móvil ya ofrecen algunos de estos servicios (por ejemplo Apple ofrece persistencia “en la nube” con iCloud), pero ha surgido una serie de compañías que ofrecen estos servicios con compatibilidad multiplataforma, de modo que por ejemplo podemos acceder a los mismos datos remotos estemos en la versión iOS o en la versión Android de la aplicación.

Dado que estamos en el módulo de persistencia, aquí vamos a centrarnos en lo que se conoce en algunos sitios como “Database as a Service”, es decir, las funcionalidades que nos permiten el almacenamiento remoto de datos (junto con las funcionalidades mínimas para autentificar usuarios en la aplicación y asignar permisos de acceso a los datos).

En la actualidad existen múltiples plataformas de terceros que ofrecen funcionalidades de Baas. Vamos a centrarnos aquí en Parse.com simplemente porque es una de las más conocidas. El resto de plataformas ofrecen funcionalidades similares.

Comenzando a trabajar con Parse

Por supuesto lo primero será registrarse como desarrollador en la web de Parse. Una vez hecho esto, debemos registrar a través del sitio web nuestra aplicación. Para cada aplicación tendremos una consola web desde la que podremos ver los logs, los datos de configuración, el estado de la base de datos, etc.

Registrar una aplicación en Parse.com

Al entrar en nuestra cuenta de Parse aparecerá nuestro dashboard, inicialmente vacío. Pulsar sobre el botón de Create new app para dar de alta una aplicación. Lo primero que nos pedirá Parse será un nombre (es simplemente una etiqueta para identificarla luego en el dashboard)

Una vez dada de alta, lo primero es obtener las claves, que vamos a necesitar cada vez que hagamos una llamada al API de Parse. En el espacio de la aplicación en el dashboard, pulsar sobre “grab your keys”. Veremos que aparecen varias claves, que son simplemente cadenas alfanuméricas:

La clave que identifica nuestra aplicación es el Application ID. En aplicaciones móviles con iOS o Android necesitamos además la Client Key. Si estuviéramos haciendo otro tipo de aplicaciones usaríamos en su lugar las otras claves (por ejemplo la Javascript key para aplicaciones web). En un momento veremos dónde usar estas claves en nuestro código.

Integrar el SDK en nuestro proyecto

Si partimos de cero, lo más sencillo es usar la plantilla de proyecto disponible en parse.com, que ya tiene todas las dependencias de los frameworks de Parse configuradas. También podemos configurar manualmente las dependencias en un proyecto ya existente siguiendo las instrucciones de la web.

Objetos y propiedades

Como se ha visto, los objetos en Parse son colecciones dinámicas de propiedades, en este sentido son muy similares a un NSMutableDictionary. Nótese que por pertenecer a la misma clase de Parse dos objetos no tienen por qué tener las mismas propiedades. Cuando un objeto no tiene una propiedad su valor es simplemente nil.

Tipos de datos

Todas las propiedades deben ser objetos, no se admiten valores primitivos:

PFObject *objetoDePrueba = [PFObject objectWithClassName:@"Prueba"];  
objetoDePrueba[@"numero"] = @42;

Pero no se admite cualquier clase Objective-C como valor de una propiedad, por ejemplo no podemos usar clases propias. Los tipos de datos válidos son PFObject (un objeto Parse puede tener como propiedad otro objeto Parse), NSNumber, NSData, NSDate, NSString, NSNull, NSArray y NSDictionary.

Intentar usar un tipo de datos distinto de los especificados como valor de una propiedad no producirá un error en tiempo de compilación, pero sí de ejecución

Leer y asignar propiedades

Para asignar una propiedad podemos usar la notación de [] o bien el método setObject:forKey

//Esto sería equivalente al ejemplo anterior  
[objetoDePrueba setObject:@42 forKey:@"numero"];

Para obtener una propiedad podemos usar la notación de [] o bien el método object:forKey

Igual que podemos añadir propiedades dinámicamente a un objeto, también podemos eliminarlas con el método removeObjectForKey

[objetoDePrueba removeObjectForKey:@"numero"];

CRUD de objetos persistentes

Ya hemos visto el método saveInBackground para almacenar objetos de forma asíncrona:

PFObject *miObjeto;  
...  
[miObjeto saveInBackground];  
//Cuidado, este código se ejecuta en paralelo al proceso de guardado  
//Seguramente el objeto todavía no esté almacenado  
NSLog(@"¿Se habrá guardado ya el objeto? ¡no creo!");

Si queremos ejecutar código una vez se haya almacenado el objeto, podemos usar saveInBackgroundWithBlock:

[objetoDePrueba saveInBackgroundWithBlock:^(BOOL succeeded, NSError *error) {  
   if (succeeded)  
      NSLog(@"¡Ahora sí sé que se ha guardado!");  
   else  
      NSLog(@"Pues no, ha habido un error: %@", error.description);  
}];

Para actualizar un objeto, cambiamos el valor de la/s propiedad/es deseada/s y volvemos a llamar a save, saveInBackground,…

Si conocemos el identificador (objId) de un objeto, podemos recuperarlo con una consulta por id. 

PFQuery *query = [PFQuery queryWithClassName:@"Prueba"];  
    [query getObjectInBackgroundWithId:@"2FtsvWLArT"  
           block:^(PFObject *obj, NSError *error) {  
        NSLog(@"%@", obj);  
}

Podemos eliminar un objeto del servidor de forma síncrona con el método delete o de forma asíncrona con deleteInBackground

Relaciones entre objetos

Para establecer una relación entre dos objetos simplemente asignamos un objeto como propiedad de otro. Veamos cómo se implementaría el ya clásico ejemplo de los blogs y los posts de un blog:

PFObject *miBlog = [PFObject objectWithClassName:@"Blog"];  
miBlog[@"titulo"] = @"Mis pensamientos más profundos";  
PFObject *unPost = [PFObject objectWithClassName:@"Post"];  
unPost[@"titulo"] = @"Hello world";  
unPost[@"blog"] = miBlog;  
[unPost saveInBackground];  
PFObject *otroPost = [PFObject objectWithClassName:@"Post"];  
otroPost[@"titulo"] = @"Hello Kitty";  
otroPost[@"blog"] = miBlog;  
[otroPost saveInBackground];

Nótese que al guardar un objeto se guardan también automáticamente y de modo recursivo los objetos relacionados. En este caso al guardar el post se guarda automáticamente el blog.

Nótese también que Parse usa referencias entre objetos, no copias, de modo que los campos “blog” de los dos posts apuntan al objeto con el mismo objectId. Cuando se guarda el segundo post ya no hace falta guardar el blog en el servidor, simplemente se referencia al objeto ya almacenado.

Como se ve, establecer relaciones uno a uno o uno a muchos es sencillo. También se pueden establecer relaciones muchos a muchos ayudándose de la clase PFRelation.

Almacenamiento local

En algunos casos puede interesarnos guardar un objeto en local pero puede no ser necesario guardarlo también en el servidor. Para uniformizar el tratamiento de los objetos y poder almacenarlos/recuperarlos de un modo similar sea en local o en el servidor, podemos activar lo que Parse denomina el “local datastore”. Este usa internamente una base de datos SQLite para almacenar la información.

Podemos activar el almacenamiento local con [Parse enableLocalDatastore] antes de fijar las claves con [Parse setApplicationId:clientKey:]. Una vez activado el almacenamiento local, podemos guardar un objeto en él con la “familia” de métodos pin, pinInBackground, …

[Parse enableLocalDatastore]  
[Parse setApplicationId:@"MI_ID_DE_APLICACION"  
       clientKey:@"MI_CLAVE_CLIENTE"];  
PFObject *unObjeto;  
...  
[unObjeto pinInBackground];

Consultas

Por supuesto se pueden buscar objetos especificando criterios de búsqueda. Vamos a ver algunos ejemplos, aunque esto es simplemente una introducción, se recomienda consultar los apartados correspondientes de la documentación de Parse.

Podemos recuperar un objeto por su id con getObjectInBackgroundWithId:block:

[query getObjectInBackgroundWithId:@"aRfx6Dz"  
       block:^(PFObject *obj, NSError *error) {  
    obj[@"prop1"] = @"unvalor";  
    [obj saveInBackground];  
}];

Más habituales serán los casos en que obtengamos no solo uno sino un conjunto de objetos resultado de aplicar una serie de restricciones. El esquema de uso general es el siguiente:

  • Crear un objeto PFQuery
  • Añadir condiciones, que pueden ser restricciones que deben cumplir los objetos u otros criterios, por ejemplo de ordenación
  • Llamar a findObjectsInBackgroundWithBlock: o findObjectsInBackgroundWithTarget:selector: para obtener un NSArray de resultados

Por ejemplo:

PFQuery *query = [PFQuery queryWithClassName:@"Alumno"];  
[query whereKey:@"edad" equalTo:@20];  
[query orderByAscending:@"nombre"];  
query.limit = 50;  
[query findObjectsInBackgroundWithBlock:^(NSArray *results, NSError *error) {  
  if (!error) {  
    NSLog(@"Encontrados %d alumnos", [results count]);  
    for (PFObject *object in objects) {  
        NSLog(@"%@", object.objectId);  
    }
  } else {  
    NSLog(@"Error: %@ %@", error, [error userInfo]);  
  }
}];

Además de usar la sintaxis que vemos aquí, para expresar condiciones también se puede usar la clase NSPredicate de Cocoa. Como veremos esta es la clase que se usa para hacer consultas en Core Data, por lo que muchos desarrolladores iOS están acostumbrados a trabajar con ella. Eso sí, Parse no implementa actualmente todas las funcionalidades de NSPredicate, únicamente un subconjunto.

Algunos detalles:

Podemos los habituales operadores de comparación como condiciones, además de equalTo tenemos notEqualTo, greaterThan, greaterOrEqualTo, …

Cuando una propiedad es un array podemos comprobar si contiene un determinado valor

//Si "notas" es un array, aquí equalTo significa "contiene" no "es igual a"  
[query whereKey:@"notas" equalTo:@10];

Con propiedades de tipo cadena podemos buscar las que contienden determinada subcadena usando la condición containsString, o buscar las que comienzan o terminan por una cadena con hasPrefix y hasSuffix, respectivamente

[query whereKey:@"nombre" hasPrefix:@"Fran"];

Cuando un objeto tiene otros relacionados no se bajan automáticamente. Por ejemplo si un objeto Cancion referencia a un objeto Disco mediante la propiedad disco y hacemos una búsqueda de canciones, al intentar acceder a esta propiedad tendremos una referencia pero no los datos del objeto. Podemos decirle a Parse que incluya los objetos relacionados que deseemos con includeKey

[query includeKey:@"disco"];

Si no nos interesara acceder al disco para todas las canciones, sino solo para algunas, podemos usar fetchIfNeededInBackgroundWithBlock con cada canción individual:

PFObject *disco = unaCancion[@"disco"];  
[disco fetchIfNeededInBackgroundWithBlock:^(PFObject *disco, NSError *error) {  
  NSString *titulo = disco[@"titulo"];  
}];

Podemos usar las relaciones entre objetos en las consultas. Por ejemplo podríamos buscar todas las canciones de un determinado disco con

PFQuery *query = [PFQuery queryWithClassName:@"Cancion"];  
[query whereKey:@"disco" equalTo:unDisco];

Usuarios y permisos

Casi todas las aplicaciones que necesitan de un backend necesitan también hacer gestión de usuarios y permisos. Necesitaremos poder registrar nuevos usuarios, hacer login y logout en la aplicación, crear objetos persistentes que sean privados de un usuario, …

Parse nos da una implementación básica de estas operaciones típicas para que podamos realizarlas de forma muy sencilla.

Gestión de usuarios

La gestión de usuarios en Parse se hace con la clase PFUser. Esta clase ya tiene unos campos predefinidos que se necesitan para hacer ciertas operaciones, como el login. Los campos que predefine Parse son email, username y password. PFUser es una subclase de PFObject, lo que significa que también podemos definir nuestros propios campos.

Podemos autentificar un usuario en una aplicación Parse con el método de clase logInWithUsernameInBackground:password::

[PFUser logInWithUsernameInBackground:@"myname" password:@"mypass"  
  block:^(PFUser *user, NSError *error) {  
    if (user) {  
        NSLog(@"Login OK!!!!");  
    } else {  
        NSLog(@"Ha fallado el login");  
    }
}];

Para hacer logout basta con llamar a [PFUser logOut] (el método es síncrono y no hay versión asíncrona).

En cualquier momento podemos obtener el usuario actual (el que se ha autentificado) con [PFUser currentUser].

Podemos registrar un nuevo usuario en la aplicación creando un nuevo objeto PFUser, rellenando los campos con la información apropiada y llamando a alguno de los métodos de instancia signUp, signUpInBackground, signUpInBackgroundWithBlock:,…

PFUser *usuario = [PFUser user];  
//Campos "estándar"  
usuario.username = @"master";  
usuario.password = @"123456";  
//Campo propio, no podemos usar la notación "."  
usuario[@"telefono"] = @"555-123456";  
[usuario signUpInBackgroundWithBlock:^(BOOL ok, NSError *error) {  
    if (ok) {  
        NSLog(@"Usuario registrado correctamente");  
    }
    else {  
        NSLog(@"Error al registrar usuario: %@", error.userInfo[@"error"]);  
    }
}];

Parse nos facilita otras operaciones típicas como confirmar un registro haciendo click sobre un enlace de validación, reinicializar una contraseña en caso de olvido, etc. Para más información consultar la documentación de Parse.

Por otro lado también podemos gestionar roles, es decir tipos de usuarios, y cada usuario puede pertenecer a uno o más roles. No obstante esto queda fuera del ámbito de esta introducción y remitimos de nuevo al lector a la documentación de Parse.

Gestión de permisos

Hemos visto que cuando se almacena un objeto persistente en el servidor, por defecto es público para lectura y escritura. Claramente esto no va a ser lo habitual en la mayoría de los casos.

Parse gestiona los permisos sobre un objeto a través de Access Control Lists o ACLs. Una ACL es una lista de permisos de lectura o escritura sobre un determinado objeto. Los permisos se pueden dar para un usuario concreto (o una lista de usuarios concretos) o para un rol (o lista de roles). Primero tendremos que crear una ACL con los permisos deseados y luego asignársela al objeto.

PFObject *obj = [PFObject objectWithClassName:@"Prueba"];  
obj[@"texto"] = @"Soy un objeto con permisos!";  
//Falta saber cómo se crea una ACL. Ahora veremos alternativas  
PFACL *miACL = ...  
//Asignamos la ACL al objeto  
obj.ACL = miACL;  
[obj saveInBackground];

En la práctica, la mayoría de los objetos van a ser totalmente privados de un usuario (solo uno los va a poder leer y modificar). El API que crea la ACL apropiada a este caso de uso es simple, basta con llamar al método de clase ACLWithUser pasándole el usuario al que queremos asociar los permisos. Por ejemplo, para crear una ACL restringida al usuario actual haríamos:

PFACL *miACL = [PFACL ACLWithUser:[PFUser currentUser]];

De modo más genérico podemos crear una ACL “vacía” e irle añadiendo permisos. Por ejemplo, podemos poner acceso público con setPublicReadAccess: y setPublicWriteAccess:(pasándole YES o NO).

También podemos controlar los permisos de lectura y escritura para un usuario individual con  setReadAccess:forUser: y setWriteAccess:forUser: 

//Creamos una ACL "vacía"  
PFACL *miACL = [PFACL ACL];  
//Esta ACL es apropiada para un objeto visible para todos  
//pero solo modificable para el usuario actual  
[miACL setPublicReadAccess:YES];  
[miACL setWriteAccess:YES forUser:[PFUser currentUser]];

Ejercicio: aplicación de mensajes con Parse

Vamos a implementar una pequeña app de chat. En principio los mensajes serán visibles por todos los usuarios autentificados, aunque también se podrían mandar mensajes privados.

Para comprobar que aparecen no solo nuestros mensajes sino también los enviados por otros usuarios, usad la misma aplicación de Parse por grupos, (por ejemplo cada mesa puede ser un grupo), es decir tenéis que usar en vuestro código las mismas claves.

Lo primero será registrarse en Parse (solo hará falta que se registre uno de cada grupo), registrar una nueva aplicación en el sitio web y obtener las claves de la aplicación. Recordad que todos los del grupo debéis usar las mismas.

En el repositorio de los ejercicios está la plantilla para no tener que comenzar de cero. Si abrís el storyboard veréis que simplemente hay dos pantallas, una para autentificarse en la aplicación y otra para ver/enviar mensajes.

Acordáos de cambiar en la plantilla las claves de la app por las de vuestro grupo. El código está en el application:didFinishLaunchingWithOptions del AppDelegate

Para poder probar la aplicación necesitaréis registrar unos cuantos usuarios. Lo podéis hacer en el dashboard de la web de Parse. En el apartado de “Core”, donde se ven los datos persistentes, podéis crear un nuevo objeto de la clase User(ya está predefinida y aparece en el panel de la izquierda con un pequeño icono de usuario al lado)

Implementad las siguientes funcionalidades:

  • Autentificación de usuarios: cuando se pulsa el botón de login en la pantalla inicial, se llama al método clickBotonEntrar: del ViewController.m. Escribir aquí el código necesario para efectuar un login con el API de Parse. Una vez autentificados correctamente, se puede seguir el segue llamado loginOK que nos lleva a la segunda pantalla con el métod performSegue de la clase ViewController.

Con esto pasamos a la segunda pantalla, la de envío/visualización de mensajes, cuyo controller es MensajesViewController

  • Envío de mensajes: cuando se escribe un mensaje en el cuadro de texto de la parte inferior y se pulsa return, se llama al método pulsadoIntroEnMensaje del MensajesViewController. Escribir aquí el código necesario para guardar el mensaje en el servidor de Parse, con el texto que se ha escrito y asociado al usuario actual.
  • Listado de mensajes:
    • Solicitar mensajes al servidor: cuando se carga la pantalla, y también cuando se pulsa el botón “Refresh” la plantilla llama al método obtenerMensajes. Introducir aquí el código que solicite al servidor la lista de mensajes ordenados por fecha. Para cada mensaje queremos que también se incluya el usuario relacionado. Ya tenéis definida en la plantilla de MensajesViewController una propiedad NSArray *mensajes para que le asignéis el array que devolverá Parse

Recordad que para que forzar a que se redibuje la tabla hay que llamar al método reloadData de la vista de tabla. En nuestro caso, en el MensajesViewController habrá que hacer [self.miTableView reloadData]

  • Mostrar mensajes en la tabla: para ello tendréis que rellenar el código de los métodos numberOfRowsInSection y tableView:cellForRowAtIndexPath:. Recordad que el primero debe devolver el número de elementos que tiene la tabla y que el segundo debe devolver la celda de la fila número indexPath.row.

  • Logout: en la plantilla, cuando se pulsa este botón se ejecuta el método clickBotonLogout. Incluir aquí el código necesario para hacer logout en Parse y volver a la pantalla anterior. Esto último lo podéis conseguir con

    [self dismissViewControllerAnimated:YES completion:nil];
